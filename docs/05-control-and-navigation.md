# Настоящие мозги: управление и навигация (ядро проекта)

Этот документ — про то, что делает косилку *умной*, а не про телеметрию.
Здесь лечатся два реальных бага MI 302:
**(A)** кручение на месте / непонимание провода и **(B)** промах мимо базы на 2–5 см.

Основано на проверенных алгоритмах open-source **Ardumower (прошивка Azurit)**:
matched-filter приёма провода, PID моторов, одометрия, IMU, docking по проводу.
Мы их адаптируем под ESP32-S3 и навешиваем сетевой слой (MQTT → HA + свой сервер).

---

## A. Приём провода: matched filter (лечит кручение на месте)

### Корень бага
Штатный приёмник = 1 катушка + порог → грубый бинарь «внутри/снаружи» без
устойчивого знака и величины. Возле провода + помехи моторов → дребезг → робот
крутится, «не понимая» направление.

### Решение
Сигнал в проводе — **асимметричный псевдошумовой код**. Приёмник коррелирует
оцифрованный сигнал катушки с этим кодом (matched filter):
- **знак пика корреляции** → внутри / снаружи (надёжно в шуме);
- **|величина пика|** → расстояние до провода (чем ближе, тем больше).

```
катушка → ОУ (усил.+смещение Vcc/2) → ADC(ESP32-S3, DMA, ~38–100 кГц)
        → matched filter (корреляция с кодом sender) → signed magnitude
        → состояние: inside/outside + дистанция до провода
```

### Требование к передатчику (sender) в базе
Знак определяется ТОЛЬКО если сигнал кодированный/асимметричный. Если штатный
sender гонит простой тон — **меняем sender** на кодированный:
ESP/Arduino + полумост (DRV8871/L298) гонит в провод код Ardumower. Дёшево,
питание от базы. (Сначала фаза 0: снять осциллографом, что именно гонит штатный.)

### Реализация на ESP32-S3
- АЦП с DMA (I2S-ADC / `adc_continuous`) — непрерывная выборка без джиттера.
- Корреляция блоком по длине кода; порог по SNR, а не по абсолюту (шумоустойчиво).
- Референс — `perimeter.h` из Ardumower (корреляция, знак, RMS). Лицензия GPL — учесть.

## B. Точная стыковка с базой (лечит промах 2–5 см)

### Корень бага
Грубое «утыкание» в провод без бокового PID и без точного довода на финише.

### Решение — слоями
1. **Wire-following c PID:** регулятор держит постоянную |величину| matched filter
   = постоянное расстояние до провода → повторяемая траектория к базе (±см).
2. **Финальные 20–40 см — мёртвый счёт:** одометрия (Холлы колёс) + IMU (курс)
   ведут прямо, без рысканья.
3. **Подтверждение по зарядным контактам:** напряжение на контактах появилось → стоп;
   не появилось на расчётной дистанции → wiggle (короткие довороты ±) до контакта.
4. **Опц. добивка до мм:** ИК-маяк на базе (приёмник на роботе) или механическая
   воронка-направляющая.

```
вход в зону базы → следование по проводу (PID на дистанции)
   → переход на guide/последний отрезок
   → одометрия+IMU прямой заход
   → детект напряжения на контактах → docked  | иначе wiggle → docked
```

## C. Архитектура мозга на ESP32-S3 (два ядра)

| Ядро | Задачи | Зависит от сети? |
|------|--------|------------------|
| **Core 0 — real-time** | ADC-DMA катушки → matched filter; одометрия (Холлы, прерывания); IMU (I2C); PID скорости моторов; конечный автомат поведения; локальный E-stop ножа | НЕТ (офлайн-safe) |
| **Core 1 — связь** | Wi-Fi; MQTT (HA + свой сервер); OTA; приём команд/расписания; выгрузка телеметрии и GPS-трека | да |

Жёсткое правило: **безопасность и базовое кошение живут на Core 0 и не зависят от
Wi-Fi/сервера**. Сервер — это «высокий уровень» (карты, погода, аналитика), не критичный путь.

### Конечный автомат поведения (минимум)
```
IDLE → MOWING → (wire hit) TURN/FOLLOW → ... → RETURN → DOCK → CHARGING → IDLE
любое состояние → E-STOP при lift/tilt/overcurrent/theft (локально)
```

## E. Почему не ESPHome для мозга

ESPHome — декларативный, отлично для телеметрии/сенсоров, но **не тянет**
matched filter + жёсткий цикл управления моторами в реальном времени. Поэтому:
- **Мозг** — кастомная прошивка C++ (PlatformIO, Arduino-ESP32/ESP-IDF), на базе Azurit.
- **Телеметрия в HA/сервер** — этот же прошивочный код публикует MQTT напрямую
  (контракт см. [01-architecture.md](01-architecture.md)). Отдельный ESPHome не нужен.

## F. Как это связано с сервером (HA + свой, оба)

- ESP32 публикует расширенную телеметрию: состояние, дистанцию до провода,
  знак (inside/outside), курс IMU, одометрию, токи, GPS-трек.
- HA: дашборд, авто-расписание по погоде, антиугон-нотификации, ручные команды.
- Свой сервер (параллельно, по тем же MQTT-топикам): карта покрытия из трека,
  аналитика (износ ножа по току, деградация АКБ), история, OTA-раскатка, PWA.
