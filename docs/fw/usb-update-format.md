# WP1 — Формат USB-обновления мейнборда (chip1, brick-safe канал)

Реверс загрузчика chip1 (`dist/gd32-mainboard-dump-v1.bin`, регион `0x08000000–0x08018000`). Метод: линейный дизасм radare2 (`pD 0x18000 @ 0x08000000`), трассировка от строк `SNK_MB_*.bin`, `firmware size error`, `SetFirmInfo`. Все адреса проверены в бинаре.

> **Главный вывод:** файл прошивки на флешке — это **СЫРОЙ образ приложения БЕЗ заголовка и БЕЗ встроенного CRC**. Версия кодируется в имени файла, размер берётся из файловой системы, CRC загрузчик считает сам и кладёт в env. → собрать прошиваемый файл и вернуть сток — тривиально.

---

## 1. Два независимых пути апдейта

| Путь | Режим USB | Кто инициатор | Формат |
|---|---|---|---|
| **A. Флешка** (наш) | USB **Host** + Mass Storage + FAT32 | косилка сама сканит диск | файл `SNK_MB_<ver>.bin`, **сырой app** |
| B. PC-тул | USB **Device** | ПК шлёт команды | стрим с 16-байт заголовком `{type,size,crc,ver}` |

Путь B (device) существует: обработчик `0x080028de` делает `memcpy(sp, rx, 0x10)` → `SetFirmInfo(type,size,crc,ver)` из 16-байтной шапки пакета, затем стримит блоки (`USB program failed`, verify `ota check file complete failed crc1/crc2`). **Нам путь B не нужен.**

---

## 2. Путь A (флешка) — точный формат

### Имя файла
`SNK_MB_<version>.bin` в корне тома (`0:/`). Матчер-глоб `0x0800366a` (строка `SNK_MB_*.bin`), парс версии из имени — `0x80071c2` (atoi по `*`). Другие типы: `SNK_BB_*`, `SNK_DB_*`, `SNK_DBL_*`, `SNK_LB_*` (под-платы), `SNK_MBTL_*`/`btl_MB_*` (**загрузчик — НЕ ТРОГАТЬ**).

### Содержимое файла
**Сырой образ приложения** — ровно то, что ложится во флеш с `0x08018000`. Заголовка/магии/CRC внутри НЕТ.
- Скан-фаза (`0x0800368a+`): из FAT-записи берёт `size = file_size` → RAM `0x20001e84`; из имени `ver` → RAM `0x20001e70`.
- IAP-фаза (`0x080030bc`, «MB IAP start»): `SetFirmInfo(type=1, size=filesize, crc=0, ver=filename_ver)`. **`crc=0`** — файл CRC не несёт.
- Программирование (`0x08003246+`): `f_open("0:/SNK_MB_..bin")` → цикл `f_read` по **1024 Б** → запись во флеш `0x8004714` (по ходу копит CRC) → финализация. Ошибки: `file open error`/`file read error`/`flash write error`/`set firmware info error`.
- CRC приложения загрузчик **вычисляет сам** (HW CRC-32) и пишет в env `MB_CRC` (+`MB_SIZE`,`mb_sv`,`MB_BVER`). Старт-селф-чек app (`fcn.08006fe8`) потом сверяет флеш с env `MB_CRC`.

### Дескриптор (`SetFirmInfo` `0x0800214c`, объект `0x20003d68`)
`+0=type +4=size +8=crc +0xc=ver +0x18=flash_base(0x18000 для MB) +0x20=env"MB_CRC" +0x24=env"mb_sv" +0x28=env"MB_SIZE" +0x2c=env"MB_BVER" +0x30=max_size(0xE8000)`.

### Гейты приёма (валидатор `0x08002216+`)
1. `size != 0`;
2. `size <= 0xE8000` (928 КБ) — иначе `firmware size error,type/size/ver`;
3. **`ver > stored_ver`** (env `mb_sv`) — иначе `no need to update, current version:%d` (**файл игнорируется!**).

Проходит → `start erase flash` → стирание региона app → `erase flash finish, start download` → программирование → `firmware %d download success`.

---

## 3. Почему это brick-safe

- **Загрузчик (`<0x18000`) при апдейте MB не перезаписывается** — мы шлём только `SNK_MB_*` (type 1). Писатель = загрузчик, он живой всегда.
- Плохой/короткий файл → ошибка IAP, старый app либо остаётся, либо не проходит старт-CRC → ребут в загрузчик → снова ждёт флешку. Худшее = «не стартует», не кирпич.
- **Единственное правило: НИКОГДА не класть `SNK_MBTL_*`/`btl_MB_*` на флешку** (это апдейт самого загрузчика, type BTL).

---

## 4. Практические следствия

### Собрать свою прошивку
1. Собрать сырой `app.bin` (линк с `0x08018000`, ≤ `0xE8000`).
2. Назвать `SNK_MB_<ver>.bin`, где `<ver>` **> текущей** установленной версии.
3. Кинуть в корень FAT32-флешки, воткнуть в USB-A мейнборда. Всё. CRC/шапку добавлять НЕ надо.

### Вернуть родную прошивку (обратимость — требование заказчика)
- **Через флешку:** вырезать из дампа стоковый app-регион (`0x08018000 .. 0x08018000+MB_SIZE`) → `SNK_MB_<оч-большая-версия>.bin` (версия должна быть выше нашей кастомной, т.к. гейт `ver>stored`). Точный стоковый `MB_SIZE`/версию уточнить (env-дамп / заголовок app).
- **Через SWD (гарантия):** прямая запись полного дампа `dist/gd32-mainboard-dump-v1.bin`. От формата и версий не зависит — работает всегда.

### Открытые пункты (доуточнить статикой)
- Точная функция финализации CRC/записи env `MB_CRC` (кандидаты `0x8004714`, `0x8008e38`, verify `0x0800469a`) — подтвердить, что путь A не сверяет CRC с файлом (он crc=0), а только пишет env.
- Стоковая версия MB (для корректного `<ver>` при откате) — из env или из app-заголовка.
- Есть ли доп. гейт по hardware version (`hv`) — в под-платах он есть, у MB проверить.
