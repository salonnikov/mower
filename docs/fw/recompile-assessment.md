# recompile-assessment.md — честная оценка пересборки/патчинга (chip1)

Вопрос заказчика: «разобрать прошивку в читаемый вид и понять, можно ли пересобрать/править». Ниже — прямой ответ без маркетинга. Легенда: `✔` — факт; `ⓗ` — оценка/гипотеза.

---

## TL;DR

- **Полный recompile из декомпила (1:1 собираемый исходник) — МИФ** для этой задачи. `ⓗ→✔ по опыту` Технически «возможно» ценой человеко-месяцев, практически — нет и **не нужно**.
- **Точечные бинарные патчи + свои модули поверх — РЕАЛЬНО.** Есть рабочий канал заливки, CRC-гейт обходится штатно (env-ключ `MB_CRC`), откат есть (полный дамп).
- **Декомпил (Ghidra/pdg) — отличный «читаемый вид» и карта для патчинга,** но это **реконструкция, а не оригинал**: имена/типы/структуры восстановлены частично.

---

## 1. Что мы реально получили (читаемый вид) `✔`

- **Дизасм:** radare2, **1778 функций** (`docs/fw/functions.md`).
- **Декомпил в псевдо-C:** Ghidra-декомпилятор через `r2 pdg` (нативный плагин в образе `radare/radare2`) — по подсистемам в `docs/fw/decompiled/`. Качество: логика читается (условия, циклы, вызовы по vtable), но:
  - имена = `fcn.АДРЕС` (символов в дампе нет; имена модулей — из `__func__`/`__FILE__`-строк);
  - структуры видны как `*(base+offset)` — типы не восстановлены;
  - крупные функции (докинг `fcn.08044188` 15.6 КБ, recover `fcn.080321d4` 11.5 КБ) декомпилируются, но с ветвлением-«простынёй».

`ⓗ` r2ghidra/pdg дал результат сразу; отдельный Ghidra headless не понадобился. Если понадобится лучшее восстановление типов — прогнать полноценный Ghidra с ручной разметкой структур (трудозатратно).

---

## 2. Почему полный recompile — миф `ⓗ`

1. **Декомпил не собирается как есть.** pdg-вывод — иллюстративный C, не компилируемый: нет объявлений структур, глобалов, vtable-типов; косвенные вызовы `(**(rN+off))()` не имеют прототипов; goto-графы.
2. **Тулчейн/линковка неизвестны.** Точные ключи компилятора, порядок секций, скрипт линкера, версии FreeRTOS/cJSON/EasyFlash — не восстановить бит-в-бит. Любое расхождение сдвигает адреса → всё, что завязано на абсолютные адреса (а тут объектный C с указателями), ломается.
3. **Объём.** ~275 КБ кода тела функций, объектная архитектура с vtable — привести к собираемому виду = переписать заново, а не «декомпилировать».
4. **Не нужно для целей проекта.** Цели (рулёжка как RC + чтение сенсоров + точечная правка докинга/навигации) достигаются патчем и своими модулями.

---

## 3. Что реально: точечные патчи `✔`

**Канал заливки существует и безопасен:**
- SWD к chip1 работает (чтение/запись RAM и флеш проверены `✔`, RDP/WRP сняты, откат = `dist/gd32-mainboard-dump-v1.bin`).
- **CRC-гейт обходится штатно:** правишь байты `0x08018000+`, пересчитываешь HW CRC-32 (poly `0x04C11DB7`, init `0xFFFFFFFF`) над `MB_SIZE&~3`, пишешь новое значение в **env-ключ `MB_CRC`** (EasyFlash). Verify проходит, кирпича нет (`fcn.08006fe8`, reverse §9). `ⓗ` перепроверить процедуру записи env на живом чипе перед первым патчем.

**Реалистичные патчи (по убыванию уверенности):**
| Патч | Куда | Уверенность |
|---|---|---|
| Зафиксировать рандом-угол съезда докинга | `fcn.08044188` (`rand_range(0x2d,0x7d)`) | `ⓗ` адрес из reverse §4, сверить |
| Сузить рандом-диапазоны разворотов | вызовы `rand_range fcn.0804f2c8` | `ⓗ` |
| Подправить `K` (угол→тик) | поля объекта движения `+0x32…+0x52` (`fcn.08020520`) | `✔` формула, поля `ⓗ` |
| Порог «near station» `0x7918` | `fcn.0804b0b8` | `ⓗ` |

**Риск патчей:** любой прямой писатель состояния/логики мимо ожидаемой точки (мы видели прямую запись `state=10` минуя `set_main_state`) → перед патчем найти ВСЕ пути (xref + watchpoint). Ошибка → ребут в загрузчик по CRC/watchdog (не кирпич, но откат нужен).

---

## 4. Что ещё реальнее и безопаснее: свои модули поверх `✔ рекомендация`

Для «рулить как RC + читать сенсоры» **бинарный патч логики не обязателен**:
- **Чтение** state/одометрии/коилов/IMU — уже делаем по SWD из RAM (анкеры подтверждены). Нужен только надёжный резолв объектов (см. odometry §5).
- **Привод** — открытый вопрос A4 бэклога: можно ли гонять моторы записью в RAM-поля/периферию по SWD малыми импульсами, ИЛИ нужен тонкий патч manual-mode. Определяется живым экспериментом, не декомпилом.
- **Внешний контур на нашем ESP** (хоминг-сенсор + доводка докинга) — единственный путь к суб-см точности; прошивка этого не умеет в принципе (reverse §4).

## 5. Вывод

| Вариант | Вердикт |
|---|---|
| Полный recompile из декомпила | ❌ миф (нереально/не нужно) |
| Читаемый вид (дизасм+псевдо-C) | ✅ есть, годен как карта |
| Точечные бинарные патчи + CRC/env | ✅ реально, канал рабочий, откат есть |
| Свои модули поверх (SWD/ESP) | ✅ основной путь под цели проекта |

`ⓗ` Все конкретные адреса патчей — перепроверить на живом чипе перед записью; флешить только с явного «да» и на стабильном питании (правило бэклога).
