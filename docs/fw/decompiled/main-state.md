# decompiled/main-state.md — главный цикл / RTOS-задачи / машина состояний (chip1)

Легенда: `✔` — подтверждено инструментом/строкой/живым замером; `ⓗ` — гипотеза, проверить на живом чипе.

Живой анкер (`✔`, SWD): **state-машина в RAM `0x200000bc`** (видели 0=idle, 2, 3=standby, 5=движение).

---

## 1. Единственная точка записи состояния: `set_main_state` = `fcn.0807976c` `✔`

verbatim pdg:
```c
void fcn.0807976c(uint32_t param_1) {
    if (param_1 < 0xb) {                 // кламп 0..10
        *(*0x8079778 + 0x10) = param_1;  // *0x8079778 = 0x200000ac; +0x10 => 0x200000bc
    }
    return;
}
```
`✔` Механически: пишет `param_1` (если `< 0xb`) в `0x200000ac + 0x10 = 0x200000bc`. Это **ровно тот адрес, что снят по SWD**. Литерал подтверждён в дизасме: `ldr r1,[0x08079778] ; =0x200000ac`.

`✔` Вызовов много (десятки xref). Пример писателя-транзишена `fcn.08025be0`:
```c
void fcn.08025be0(uint param_1) {
    ...
    if (iVar1 == -1) { ...; fcn.0807976c(9); ... return; }  // set_main_state(9) = докинг
    *(*0x8079778 + 0x10) = 10;                              // ПРЯМАЯ запись state=10 (зарядка)
}
```
`✔` **Важно:** часть кода пишет `0x200000bc` **напрямую** (минуя `set_main_state`), напр. `=10`. Значит при патче логики нельзя полагаться только на перехват `fcn.0807976c`.

## 2. Таблица состояний

Значения-якоря `✔` (живой замер): `0=idle`, `2`, `3=standby`, `5=движение`. Остальное — `ⓗ` из строк логов транзишенов (см. `../firmware-gd32-reverse.md` §2; там разметка 0..11). Кламп в `set_main_state` = `0..10` (`✔`), значит `11` не может быть main_state — это отдельный stop/pause-уровень `ⓗ`.

| Состояние | Смысл `ⓗ` | Опора |
|---:|---|---|
| 0 | idle/готова | `✔` живой замер = 0 |
| 2,3 | boot/standby | `✔` живой замер (2,3) |
| 5 | движение/transitional | `✔` живой замер = 5 |
| 6..10 | idle6/error7/кошение8/докинг9/зарядка10 | `ⓗ` из строк логов и прямых записей (`=10` в `fcn.08025be0`) |

> ⚠️ Нумерация 6..10 взята из лог-строк переходов, не из живого замера — держать как `ⓗ` до сверки по SWD во всех режимах.

## 3. Boot / запуск планировщика `✔` (механически)

`✔` Reset-обработчик `fcn.08011a3c` (в области загрузчика, `< 0x18000`). Финальный шаг — `fcn.080003bc`:
```c
void fcn.080003bc(void) {
    (**0x80003ec)();   // ptr1: включение FPU (CPACR|=0xF00000) + пролог, ⓗ
    (**0x80003f0)();   // ptr2: xPortStartScheduler (планировщик забирает управление), ⓗ
    return;
}
```
`✔` Механически: два косвенных вызова по указателям из литерал-пула. `ⓗ` Назначение (FPU-enable + старт FreeRTOS-планировщика) — из контекста (`__FILE__`-строки `platform/freertos/port.c`) и того, что после него код не возвращается.

## 4. RTOS-задачи (FreeRTOS) `✔`

`✔` Подтверждено строками создания задач (`izz`, «create … task failed»):

| Задача | Строка-опора (vaddr) |
|---|---|
| **process manager** (мастер-машина) | `0x08054c97` "create process manager task falied" |
| time | `0x08055cc6` |
| **bms** | `0x08056c9b` |
| **bdport** init/send (граница/провод) | `0x08056e2f` |
| **dpport** init/send/send-log (дисплей/ESP) | `0x080570d3` / `0x0805893b` / `0x08058a73` |
| **ledport** init | `0x080572b7` |
| map (картирование) | `0x0805843b` |
| **highspeed loop** | `0x08058663` |
| **mems** init (IMU) | `0x0805ac87` |
| fuse (сенсор-фьюжн) | `0x0805b26b` |

`ⓗ` Мастер-машина состояний крутится в задаче **process manager**; её тело статикой полностью не выделено (r2 не создал функцию по xref строки — область частично помечена DATA). Точку входа задачи достоверно даст живой брейк на `set_main_state` или чтение TCB по SWD.

`✔` Планировщик — FreeRTOS heap_4, порт ARM-CM4F (`__FILE__` `platform/freertos/{tasks,queue,port,heap_4}.c`). Очереди/мьютексы (строки): «left/right wheel speed queue», «speed data queue», `motor_spi_mutex`, control mutex, «dpport send lock mutex», `pnt_queue`.
