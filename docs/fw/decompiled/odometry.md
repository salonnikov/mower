# decompiled/odometry.md — одометрия колёс (chip1, GD32F305)

Живые анкеры (проверены по SWD, `✔`): два симметричных счётчика **`0x2000be28` (лев)** и **`0x2000be58` (прав)** тикают при езде. Задача этого файла — механически показать код вокруг них и честно отделить доказанное от гипотез.

Легенда: `✔` — подтверждено выводом инструмента/строкой/живым замером; `ⓗ` — гипотеза, проверить на живом чипе.

---

## 1. Ключевой факт про адресацию счётчиков (важно для верификации)

`✔` **В коде НЕТ ни одного абсолютного литерала `0x2000be28` / `0x2000be58` / `0x2000b250`.** Проверено полным поиском по образу:
- поиск литерал-пулов `/x 28be0020`, `/x 58be0020`, `/x be0020` → **0 совпадений**;
- поиск в полном линейном дизасме приложения (`pD 0xe8000 @ 0x08018000`) immediate’ов `0xbe28/0xbe58/0x2000be` → **0 совпадений**.

`✔` Вывод: эти адреса **не статические глобалы, загружаемые литералом**, а **поля структуры, доступные через базовый указатель** (`ldr rBase,[pool]; str rX,[rBase+off]`). Именно так в этой прошивке адресуются все объекты (ср. `set_main_state`: грузит базу `0x200000ac`, пишет в `+0x10` → `0x200000bc`).

`ⓗ` Следствие для реверса: **точную функцию-писателя `0x2000be28` статикой не приколотить** без резолва рантайм-значения базового указателя. Достоверно её укажет только **живой аппаратный watchpoint по SWD на запись `0x2000be28`** (см. раздел 5).

## 2. Косвенное, но сильное структурное свидетельство: шаг 0x30

`✔` Счётчики разнесены на `0x2000be58 − 0x2000be28 = 0x30` (48 байт).

`✔` В инициализаторе объекта движения `fcn.0805640c` (создаёт «left/right wheel speed queue», см. ниже) ровно этот шаг `0x30` фигурирует как размер под-структуры колеса:
```c
// fcn.0805640c (verbatim pdg, фрагмент)
iVar1 = *(iVar4 + 4);
*(iVar1 + 8)   = iVar1 + 0x30;   // указатель на «второе» под-поле, +0x30
*(iVar1 + 0xe4)= iVar1 + 0x30;
iVar1 = fcn.0802b6fc(0x1e,2);     // xQueueCreate(30,2) — очередь скорости колеса
*(*(iVar4 + 4) + 0xe8) = iVar1;   // "create left wheel speed queue failed" @0xe7
...
iVar1 = fcn.0802b6fc(0x1e,2);     // вторая очередь
*(iVar4 + 0xec) = iVar1;          // "create right wheel speed queue failed" @0xed
```
`ⓗ` **Гипотеза:** одометрические счётчики `be28`/`be58` — это **одноимённое поле в двух под-структурах колеса, разнесённых на 0x30**, внутри объекта движения (тот, чей указатель лежит в `*0x8056780` / `*0x80564c0`). Симметрия «лево/право» и шаг 0x30 совпадают идеально, но привязку базы к `0x2000be00` нужно подтвердить живьём.

Доказательства строк (`izz`, `✔`):
- `0x08056544` `"create left wheel speed queue failed"`
- `0x0805656c` `"create right wheel speed queue failed"`
- `fcn.0802b6fc(0x1e,2)` = `xQueueCreate(len=0x1e=30, itemsize=2)` `ⓗ` (сигнатура FreeRTOS).

## 3. Коэффициент тик → мм (калибровочная цепочка)

`✔` В образе есть готовая формула геометрии робота (строки, `izz`):
- `0x080563dc` `"wheel perimeter=%d, redution ratio=%g error"`
- `0x0808b4b8` `"robot shape wheel dis=%d,wheel perimeter=%d,redution ratio=%g,blade diameter=%d,blade motor polar=%d, creep motor polar=%d, cutting speed=%d, turbo speed=%d,cycle radius=%d"`
- `0x08077780` `"wheel_dis"`, `0x08085f0b` `"set_wheel_perimeter"`, `0x08085f1f` `"set_wheel_dis"`
- `0x0807d454` `"set robot wheel distance =%dmm failed, err=%d"`

`ⓗ` **Гипотеза (пересчёт):** линейный путь колеса `дистанция_мм = тики × wheel_perimeter_mm / (counts_per_rev × reduction_ratio)`, где:
- `wheel_perimeter` — длина окружности колеса (мм), конфиг-ключ, задаётся `set_wheel_perimeter`;
- `reduction_ratio` — редукция мотор→колесо (тип `%g` — float);
- `counts_per_rev` — тиков энкодера на оборот мотора `ⓗ` (в строках не выведен; замерить живьём).

`ⓗ→калибровка` Практика: **не выводить коэффициент из прошивки, а замерить** — прокатить/прокрутить колесо на известное расстояние (пользователь крутит рукой, задача A2 бэклога), снять Δтиков по `0x2000be28`, получить мм/тик напрямую. Это надёжнее, чем доверять реконструированной формуле.

## 4. Формула угла разворота → тики (смежная, но подтверждённая) `✔`

`angle_to_cycle_tick` = **`fcn.08020520`**, verbatim pdg:
```c
uint32_t fcn.08020520(int param_1) {   // param_1 = угол (градусы)
    int iVar1 = *(*0x802057c + 4);      // объект движения
    int iVar3 = *(iVar1 + 4);           // режим/модель (0..4)
    uint32_t uVar2;
    if      (iVar3 == 0) uVar2 = *(iVar1 + 0x32);
    else if (iVar3 == 1) uVar2 = *(iVar1 + 0x3a);
    else if (iVar3 == 3) uVar2 = *(iVar1 + 0x4a);
    else if (iVar3 == 2) uVar2 = *(iVar1 + 0x42);
    else if (iVar3 == 4) uVar2 = *(iVar1 + 0x52);
    else { /* лог ошибки 0x3df */ return 0; }
    return (param_1 * 0x2b9) / uVar2;    // угол × 697 / K
}
```
`✔` Механически: `тики = угол × 0x2b9(=697) / K`, где `K` — пер-модельная константа из полей объекта движения `+0x32/+0x3a/+0x42/+0x4a/+0x52` (выбор по полю `+4`). Разворот исполняется разомкнуто в тиках. `ⓗ` Смысл 697 и K — эмпирические константы шасси (проверить сверкой с реальным поворотом).

## 5. Что осталось `ⓗ` — план подтверждения живьём

| Вопрос | Как проверить на живом чипе |
|---|---|
| Кто пишет `0x2000be28`/`be58` | SWD **watchpoint на запись** `0x2000be28` → OpenOCD `bp/wp`; посмотреть PC писателя, сопоставить с `afl`. Ожидание `ⓗ`: периодический ридер энкодера (TIMERx encoder mode) или ISR. |
| Через какой TIMER считаются тики | После watchpoint — проверить, читает ли писатель `TIMERx_CNT`. Кандидаты encoder-mode: TIMER1/2/3/4 (см. `memory-map.md`). |
| Базовый указатель объекта движения | Прочитать `*0x8056780` и `*0x80564c0` по SWD в активном состоянии; проверить, попадает ли `база+off == 0x2000be28`. |
| Коэффициент тик→мм | Замер A2: известное расстояние ↔ Δтиков. Не доверять реконструированной формуле раздела 3 без замера. |
