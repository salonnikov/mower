# WP2 + WP3 — контракт драйверного API и точка инъекции (по полному Ghidra-декомпилу)

Итоговый разбор для Варианта 1 (тонкий app-исполнитель на chip1, RC-управление снаружи). Опора — `docs/fw/ghidra/chip1-decompiled.c` (1838 функций). `✔`=прочитано в декомпиле/резолвнуто из бинаря; `ⓗ`=требует финальной проверки на живом чипе.

---

## 1. Единая архитектура (подтверждена везде)
Объектный C: сервисы = структуры с **vtable** (указатели методов), вызовы `(**(obj+off))()`. Значения датчиков кэшируются в полях объекта, читаются геттером vtable. → **наш модуль держит указатели на сервис-объекты (фикс. RAM) и зовёт их методы**; это переживает пересборку/версию, не завязано на абсолютные адреса тел функций.

| Что | Функция | Контракт |
|---|---|---|
| Левое колесо | `left_motor_rotation` ~`0x0805f620` | `(obj=lmdrv, speed)`; `obj=*0x2000054c`; speed 0..6141 → PWM=`speed*0x1c20/K`, `K∈{1533,3069,6141}` полосами, кламп PWM `718..7200` |

| A4963 SPI запись+verify | `FUN_0801f4dc(obj, obj+0x30, dir)` | dir∈{0,1,2}→биты `obj+0xa (&0xff8f|0x10/0x20/0x30)`; verify через `FUN_080840d8`(wr)/`FUN_0806caac`(rd), ≤49 ретраев |

## 4. WP3 — точка инъекции `✔ анкеры`
Мастер-машина — **сервис process-manager** (конструктор `FUN_0804ef44`):
- **Слот синглтона: RAM `0x20000078`** (`*0x20000078` = указатель на обёртку сервиса).
- Обёртка: `[0]`=метод0 `0x080549f0`, `[1]`=метод1 `0x080706a0`, `[2]`=ctx-объект (0xb8 байт, heap).
- Главное состояние (то, что снифается): байт `0x200000bc` (= `*(0x200000ac)+0x10`), пишется только `set_main_state 0x0807976c` (клампит 0..10).

**Стратегия инъекции (по возрастанию инвазивности):**
1. **Перехват tick-метода:** после boot переписать `(*0x20000078)[0]` (метод0) на адрес нашей функции. Наш tick: читает внешний UART-команд → зовёт `*_motor_rotation` + драйверные геттеры → шлёт телеметрию. RTOS/драйверы/watchdog живут как есть. Наименее инвазивно, обратимо.
2. **Своя FreeRTOS-задача** + перевод process-manager в «idle»/no-op (не пускать его в навигацию). Кормёжку watchdog (IWDG `0x40003000`) держат несколько задач — наш tick тоже должен кормить, если гасим родные.
3. Внешний протокол: переиспользуем родной JSON-фреймер `&{…}<crc8>#` (CRC-8 `FUN_080241d0`) — не писать транспорт с нуля.

**Watchdog:** IWDG активен, кормится из нескольких задач; при подмене логики убедиться, что кормление продолжается (иначе reset — не кирпич, но перезагрузка).

## 5. Статус задач
- **WP1** ✅ формат USB brick-safe (`usb-update-format.md`).
- **WP2** ✅ контракт приводов+датчиков (эта таблица; ток/дождь/кнопки/нож/IMU — пути подтверждены, часть офсетов `ⓗ` до живой проверки).
- **WP3** ✅ анкеры инъекции определены (слот `0x20000078`, метод0 `0x080549f0`, состояние `0x200000bc`).
- **WP4/WP5** (сборка `arm-none-eabi`+линк `0x08018000`+упаковка `SNK_MB`; заливка) — это уже **выход за статический анализ**, делается когда будет железо/добро на сборку. Формула файла и канал известны (WP1).

`ⓗ` Все офсеты/адреса — финально подтвердить на живом чипе (SWD-watchpoint/чтение объектов) перед первой сборкой. Абсолютные адреса тел функций уточнять по декомпилу; вызовы драйверов — предпочтительно через vtable сервис-объектов, не по жёстким адресам.
