# Архитектура прошивки косилки MI 302 — сводная карта (chip1 + chip2)

Единый разбор по декомпилированным файлам `docs/fw/` (chip1) и `docs/fw-chip2/` (chip2) + живые SWD-находки (`docs/live-findings.md`). Правило: **✔ = подтверждено (код/строка/живой замер), ⓗ = гипотеза (проверить на живом чипе).** Ничего не выдаём за факт без пометки.

---

## 0. Система в целом

Две платы, три MCU:
- **chip1 = GD32F305** (Cortex-M4F, 1 МБ флеш, FreeRTOS) — **главный мозг**: моторы колёс, нож, зарядка/BMS, IMU, машина состояний, навигация, все UART-линки. ← сюда патчим/рулим.
- **chip2 = GD32F30x** (Cortex-M4, FreeRTOS) — **плата коилов**: два носовых индуктивных датчика → ADC → магнитуда/сторона → JSON в chip1 по USART2. Моторами НЕ рулит. Фолтит на SWD-halt (busy-wait по EOC ✔).
- **Дисплейная плата (ESP32+ARM)** — UI/кнопки (вкл/старт/домой/ок), пишет на chip1 по дисплейному UART тем же кадром.

Связь: `& JSON <CRC8> #` кадры по UART между платами (chip1 ↔ дисплей ↔ bdport). chip2→chip1 — **голый JSON без рамки/CRC**.

## 1. chip1 — карта флеша и загрузка

| Регион | Что | ✔/ⓗ |
|---|---|---|
| `0x08000000–0x08018000` | **Загрузчик (BTL)** 96 КБ; reset `fcn.08011a3c` | ✔ |
| `0x08018000–0x08100000` | **Приложение (MB)** ≤0xE8000; CRC-гейт над `0x08018000` | ✔ |
| Внешняя SPI-флеш | EasyFlash env (`MB_CRC/MB_SIZE`) + OTA staging | ⓗ |

**Boot:** reset `fcn.08011a3c` → `fcn.080003bc`: включение FPU + старт FreeRTOS-планировщика (`platform/freertos/port.c`) ✔.
**CRC-гейт прошивки:** `fcn.08006fe8` считает HW CRC-32 (`0x40023000`, poly `0x04C11DB7`, init `0xFFFFFFFF`) над приложением, сверяет с env `MB_CRC`; не сошлось → reset в загрузчик. Обход: 1 байт `0x080070d4` `beq→b` (`D0→E0`) — гейт всегда пройден (найдено, байт сверен с живой флешкой ✔). RDP/WRP сняты, откат = полный дамп ✔.

---

## 2. chip1 — RTOS-задачи (FreeRTOS heap_4, порт CM4F) ✔

Подтверждены строками создания задач:
`process manager` (мастер-машина состояний), `time`, `bms`, `bdport` (граница/провод), `dpport` (дисплей/ESP), `ledport`, `map`, **`highspeed loop`** (контур моторов), `mems` (IMU), `fuse` (сенсор-фьюжн).
Очереди/мьютексы: «left/right wheel speed queue», «speed data queue», `motor_spi_mutex`, control mutex, «dpport send lock mutex», `pnt_queue`.
> Тело `process manager` статикой r2 не выделил (область как DATA) — точку входа даст брейк по SWD ⓗ.

---

## 3. chip1 — машина состояний ✔

- Поле state = **`0x200000bc`** ✔ (живой замер). Единственный сеттер `set_main_state fcn.0807976c` (кламп 0..10, пишет в `*0x200000ac + 0x10`). ⚠️ **Есть прямые записи state минуя сеттер** (напр. `=10` зарядка в `fcn.08025be0`) — при патче логики нельзя перехватывать только сеттер ✔.
- Значения (✔ живьём): `0`=idle (сервис-объекты NULL), `2/3`=standby (объекты подняты), `5`=движение, `8`=подготовка/departure (наблюдали). `6..10` (idle6/error7/кошение8/докинг9/зарядка10) — ⓗ из лог-строк. **E9 = не дошла до провода** (нав.ошибка) ✔.

---

## 4. chip1 — приём команд (UART) ✔

Три одинаковых JSON-линка: **dpport** (дисплей/ESP), **bdport** (граница), **ledport**.
Кадр: `&`(0x26) + JSON + CRC-8/MAXIM(1B, poly 0x8C, `fcn.080241d0`) + `#`(0x23). Фреймер `fcn.0801fb80`, дефреймер `fcn.0806e938`.

**Диспетчер (dpport `fcn.080470f0`):** `cJSON_Parse` → `GetObjectItem("cmd")` → `category = valueint & 0xFF000000` → линейный поиск в таблице `[+0x60]=category / [+0x64]=handler` (**16 слотов**) → `handler(json)`. Регистрация хендлеров: `fcn.08020036(category, handler)`. bdport/ledport — та же логика, таблица `+0x50/+0x54`, **5 слотов**.
- `cJSON_Parse`=`fcn.08023fe6`, `GetObjectItem`=`fcn.08023f80` (`item+0x14`=valueint, `+0x10`=valuestring). ✔
- **STATUS-репорт (TX, MB→дисплей): cmd `0x50000000`**, строит `fcn.08081244` инкрементально (поля state/error/bat_lv/bat_per/rain/stop/border/station/area/minutes/temp…) ✔.
- `0x33000020` = запрос STATUS от дисплея → триггерит репорт ⓗ (регистрация не найдена статикой).
- **Конфиг `fcn.08076b2c`**: плоский разбор **~110 ключей** (`GetObjectItem(key)→valueint/valuestring→cfg`). Важные: `perimeter, ratio, wheel_dis, blade_diameter, bmotor_polar, cmotor_polar, blade_sp, blade_max_sp, border_offset, station, shape_param` + `*_en`-флаги (`lift_en, border_en, mems_en, smooth_en, mtrack_en, qdock_en, pwd_en, blade_check_en…`). ✔
> Готовой команды «ехать/jog/motor-test» НЕТ — в мастер-декодере команд движения только start/trim/dock/poweroff (bit-маска), см. reverse. Для RC нужен новый хендлер/патч.

## 5. chip1 — ПРИВОД (главное для управления) ✔/ⓗ

Двухуровневый: движ.-команда → расчёт PWM → highspeed-loop → драйвер мотора.

**set_wheel_speed, два канала** (разные контексты!):
- **Канал A `fcn.0805f620`**: контекст `*0x805fa18 = 0x2000054c`. «Коробка передач»: по скорости `arg1` выбирает передачу 0/1/2 (пороги `0x266/0x99/0x132/…`), при смене передачи зовёт `fcn.0801f4dc` **«brake to dir»** (смена направления с verify-циклом), затем `PWM = speed*0x1c20/gear_max` (gear_max ∈ {`0x17fd`=6141, `0xbfd`=3069, `0x5fd`=1533}), насыщение до `0x1c20`=7200. Выход: **`fcn.0807f696(*0x805fac4, 3, pwm)`** (запись PWM в канал регистра). ✔
- **Канал B `fcn.08020e0c`**: контекст `*0x805fb9c`, регистр `*0x805fba4`. Скорость зажата 500..4000, `PWM=(speed*7200)/5500` (`0x157c`), знак через `fcn.0805f3d8`. Выход тот же `fcn.0807f696(...,3,pwm)`. ✔
- Строки: `"max speed=6141, brake to dir=%d, speed=%d over max, pwm=%d"` ✔.
- Угол→тик: `fcn.08020520`: `tick = angle*697/K` (K — пер-модельная из полей объекта движения `+0x32..+0x52`) ✔. Разомкнуто (без ОС).
- Команда «ехать arg1=скорость, arg2=угол»: `fcn.0804bb0c` (через vtable драйвера: скорость, тики, пуск) ⓗ.

**⚠️ ЖИВОЙ ВЫВОД (✔, критично):** статическая запись в CCR PWM колёса **НЕ крутит** — колёса встают через ~1.5с несмотря на CCR=4000. Моторы почти наверняка **BLDC/коммутируемые прошивкой** (`brake to dir` = смена направления/фазы). Значит **рулить можно только через drive-функции прошивки** (`fcn.0805f620/08020e0c` / `fcn.0804bb0c`), а не поком регистра. Мой прежний «CCR-руль» = ошибка.
- Регистр PWM (какой TIMER) = аргумент `*0x805fac4`/`*0x805fba4` в `fcn.0807f696` — резолвить живьём. memory-map: advanced **TIMER0 `0x40012c00`/TIMER7 `0x40013400`** = кандидаты BLDC-PWM, SPI0 `0x40013000` под `motor_spi_mutex` ⓗ. (TIMER2 `0x40000400`, что я тыкал, — вероятно НЕ колёсный привод.)

## 6. chip1 — одометрия ✔ (адрес) / ⓗ (писатель)

- Счётчики: **`0x2000be28` (лев) / `0x2000be58` (прав)**, шаг `0x30` ✔ (живой замер, тикают при езде).
- В коде НЕТ абсолютного литерала этих адресов → это поля структуры через указатель (объект движения, init `fcn.0805640c` создаёт «left/right wheel speed queue», под-структуры колеса шагом `0x30`) ✔ структурно, ⓗ привязка базы.
- Писатель = ⓗ (нужен SWD-watchpoint на запись `0x2000be28` + чтение `TIMERx_CNT`). Кандидаты encoder-mode: TIMER1/2/3/4.
- Тик→мм: **замерить** (A2), не выводить из прошивки. Конфиг: `set_wheel_perimeter/set_wheel_dis/ratio`.

---

## 7. chip1 — IMU (курс/наклон) ✔

- Датчик: **BNO055** (Bosch, `driver_mems_snk_v13`, chip-id `A0 FB 32 0F` по I2C ✔; альтернатива TDK ICM-42688 для др. моделей, выбор по коду платы ⓗ). I2C1 `0x40005800` ⓗ.
- Углы: объект `*(0x2000039c)`, поле `[+4]`: **heading `+0x18`, roll `+0x1a`, pitch `+0x1c`** (мгновенные), avg `+0x1e/+0x20/+0x22`. **Единицы = целые градусы** ✔ (пороги 360/180/90). heading 0..359, калибр.смещение `[+0x3e]`.
- Обновление `fcn.0803a198` (периодич.), усреднение `fcn.0802b868` (pitch сглаж.24:1, heading (old+new)/2, roll без сглаж.).
- **SLOPE-детект `fcn.08039fc4` использует ТОЛЬКО pitch+roll, heading НЕ читает** ✔. Валидатор `fcn.08028c70`.

## 8. chip1 — коилы/граница (приём от chip2) ✔

- Дискрет стороны в контексте узла границы: **`*(ctx+0x24)`=ЛЕВО, `*(ctx+0x2c)`=ПРАВО**, значения 0/1/2 ✔. Сырой бит `fcn.0804f2ee` (1/2). Резолвер `fcn.0802c424` (пара+режим`+0x12a`→0/1/2). Событие пересечения `fcn.0802c900`→callback навигации `vtable+0x88` ✔.
- **near-station: обе магнитуды ≤ `31000` (0x7918)** + валидны → близко к базе `fcn.0804b0b8` ✔. Сигнал: `fcn.0805034c()`→`vtable+0x10`→`{+4:magL,+8:magR,+0xc:okL,+0x10:okR,+0x14:state}` ⓗ.
- Сырьё: у chip1 тоже есть ADC0 (`0x40012400`)+DMA0 сканер каналов {9,11,12,13,14,15(+16,17)} в буфер `*(0x200005dc)/*(0x20000620)` ⓗ (какой канал=катушка — живьём). Основной источник ls/rs — всё же chip2 по JSON.

## 9. chip1 — нож ✔

- Сервис `fcn.08022938` (force blade fwd/back). Объект `*0x8022a00`, `[+4]`=ctx: `+8`=скорость, `+0x10`=vtable драйвера, `+0x25`=запрош.направление, `+0x27`=тек.направление.
- **Точки:** `ops+0x14 = set_forward(speed)`, `ops+0x18 = set_backward(speed)` ✔. Драйвер `fcn.08022a80` (`driver_blade_motor_snk_v2`), enable GPIOC15 ⓗ. Скорость ножа из конфига `blade_sp/blade_max_sp`.

## 10. chip2 — плата коилов ✔

- **ADC0 `0x40012400` ch5 = LEFT, ADC1 `0x40012800` ch9 = RIGHT**, sample-time 7, синхронный dual (`fcn.08019bf4`) ✔. Триггер TIMER1 `0x40000000` TRGO, DMA0 `0x40020000` → буфер `[obj0x20000078 +0x70]`, число отсчётов `[+0x92]` (~800 ⓗ).
- **Базовая линия** = среднее 8 отсчётов → **`0x2000004c`**, норма 1906..2189 (~2048), иначе default 2048 (`fcn.0801a1f8`) ✔.
- Обработка волны (`~0x080159a0–0x08016900`): сегментация по нулям, фичи сегмента (ширина 4..19 ✔, пик-амплитуда=магнитуда), классификатор in/out (коды 0/1/2). `follow_border_check_wave fcn.080164bc`, `follow_check_subwave fcn.08016784`, lift `deal_lift_sensor fcn.08013970` (дебаунс 10).
- **Отправка в chip1: `send_border_message` вход `0x080195c4`** — голый cJSON по USART2 `0x40004800`, без рамки/CRC: `{"cmd":16,"ls":N,"rs":N,"lf":N,"sls":N,"srs":N}` ✔ (cmd=float16.0, ключи через cJSON_AddNumberToObject, знак=сторона). `lf` из объекта `0x20000070`. Линк **односторонний** (нет cJSON_Parse ✔).
- Соседние: `send_search/follow/version_message`.

## 12. Что нужно для «своей прошивки»/управления — выводы

1. **Полный recompile из декомпила = миф** (не нужно). Путь: **точечные байт-патчи + свои модули**. Канал заливки рабочий, откат есть.
2. **Рулить = только через drive-функции прошивки** (BLDC-коммутация обязательна; поком регистра — нет). Два реальных варианта:
   - **(A) SWD-вызов** `fcn.0805f620/08020e0c(speed)` (или `fcn.0804bb0c(speed,angle)`) с малины — halt+call, проверить, что коммутация идёт; риск конкуренции с highspeed-loop.
   - **(B) Патч manual-mode**: новый хендлер команды (свободный слот в таблице dpport, регистрация `fcn.08020036`) → в моторном состоянии гонит `set_wheel_speed` с нашими L/R, без border-навигации (обойти гейт в `deal_start_command`). Малина шлёт команду по UART **или** пишет mailbox по SWD.
3. **Чтение сенсоров — уже умеем** (state/одометрия/IMU/коилы из RAM по SWD; ls/rs также из JSON chip2).
4. **Точная навигация/докинг** — только свой внешний контур (прошивка суб-см не умеет).

## 13. Открытые ⓗ (нужен живой чип, без halt для chip2)

- Писатель одометрии `be28/be58` + через какой TIMER (watchpoint). Тик→мм (замер A2).
- Какой TIMER/регистр реально = колёсный BLDC-PWM (резолв `*0x805fac4`/`*0x805fba4`). Левый/правый по каналам A/B.
- Идёт ли коммутация при SWD-вызове `set_wheel_speed` (тест варианта A).
- Физический ADC-канал ↔ левая/правая катушка; адрес RAW-буфера (`*0x200005dc`/`*0x20000620`).
- Точная семантика state 6..10, все прямые писатели `0x200000bc`.
- chip2: PSC/ARR TIMER1, размер буфера `[+0x92]`, GPIO-пины, окончательное подтверждение односторонности RX.
