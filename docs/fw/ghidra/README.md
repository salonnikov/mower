# Полная декомпиляция обоих процессоров (Ghidra headless)

Читаемый псевдо-C по всей прошивке обоих чипов косилки. Получено **Ghidra 12.1.2 headless** (полноценный анализ + декомпилятор), не r2 — Ghidra лучше восстанавливает границы функций, включая вызываемые только через vtable.

## Файлы
| Файл | Что |
|---|---|
| `chip1-decompiled.c` | chip1 (мозг), **1838 функций, 0 неудачных** декомпиляций |
| `chip2-decompiled.c` | chip2 (датчик провода), **420 функций, 0 неудачных** |
| `chip1-functions.idx` / `chip2-functions.idx` | индекс: `FUN_<адрес> @ <адрес> size=<байт>` |
| `DecompileAll.java` | Ghidra-скрипт экспорта (для воспроизводимости) |

Имена функций — `FUN_<адрес>` (символов в дампе нет). Навигация: адрес из наших доков (`0x08044188` и т.п.) → `FUN_08044188` в файле/индексе.

## Процессоры (под них подходит код)
- **chip1 — GigaDevice GD32F305, ARM Cortex-M4F @120 МГц, FPU.** Флеш 1 МБ @`0x08000000` (загрузчик 96 КБ + приложение @`0x08018000`), SRAM @`0x20000000`. FreeRTOS. Язык Ghidra: `ARM:LE:32:Cortex`.
- **chip2 — GD32F30x, ARM Cortex-M4.** 256 КБ, база `0x08000000`, тот же язык.

Обе прошивки импортированы **как raw binary с базой `0x08000000`** (векторы/reset совпали независимо, см. `../../firmware-gd32-reverse.md`).

## Воспроведение (в docker, без установки на хост)
```
# Ghidra 12.1.2 + eclipse-temurin:21-jdk
analyzeHeadless <proj> chip1 \
  -import gd32-mainboard-dump-v1.bin \
  -processor ARM:LE:32:Cortex -loader BinaryLoader -loader-baseAddr 0x08000000 \
  -scriptPath <dir> -postScript DecompileAll.java -deleteProject
# env DECOMP_OUT=<out.c>; аналогично для chip2 (chip2-dump)
```

## Перекрёстная валидация с r2
Крупнейшая функция chip1 — `FUN_08044188` = **15168 байт** = мастер докинга (r2 давал `0x08044188`, 15.6 КБ). CRC-8, `set_main_state`, приводы, докинг — сходятся между Ghidra и r2 (см. `../verification-log.md`).

## Как это использовать
- **Читаемая карта** для патчинга и понимания логики (докинг/навигация/BMS/протокол).
- **Не 1:1-рекомпилируемый исходник:** это реконструкция (нет типов структур, имена `FUN_*`, косвенные vtable-вызовы). Для нашей цели (Вариант 1 — тонкий app-исполнитель) не требуется 1:1: см. `../recompile-assessment.md`, `../driver-api-map.md`.
- **Заливка/откат** — brick-safe через USB `SNK_MB_*.bin`, см. `../usb-update-format.md`.

## Дальше по этому декомпилу (WP2 остаток / WP3)
Теперь оставшиеся геттеры и офсеты vtable добираются **грепом по этим файлам**, без угадывания адресов:
- right-motor: искать функцию с множественными вызовами `FUN_0801f4dc` (A4963 SPI) в районе `0806f…`.
- IMU-курс/одометрия/УЗ: греп по строковым переменным (`s_..._<addr>`) и по конструкторам сервисов (регистрация vtable).
- WP3-инъекция: `process manager` — задача и сервис-объект (диспетч `0x200000bc`).
