# Навигация: гибрид «провод периметра + GPS»

Выбранная стратегия: **едем по штатному проводу периметра** (надёжно, дёшево,
работает из коробки), а **GPS добавляет «ум» поверх** — трек, карту покрытия,
гео-зону, антиугон и аналитику. GPS НЕ управляет движением на этом этапе.

## Разделение ролей

| Задача | Кто отвечает |
|--------|--------------|
| Удержание в границах | Штатный провод периметра + родной MCU |
| Защита lift/tilt/bump | Родной MCU + дубль на ESP32 |
| Где робот сейчас (трек) | GPS на ESP32 |
| Карта покошенного | Сервер: пост-обработка GPS-трека + одометрии |
| Антиугон / гео-зона | ESP32 (детект) + сервер (тревога) |
| Расписание/погода | Сервер |

## GPS-данные и формат

ESP32 читает NMEA с UART GPS, парсит (TinyGPS++ / встроенный парсер ESPHome) и
публикует в `mower/mi302/gps`:
```json
{ "lat": 55.751244, "lon": 37.618423, "fix": 1, "sats": 9, "hdop": 0.9, "speed": 0.3 }
```
- `fix=0` нет решения, `1` 2D/3D. При `fix=0` — не строить трек, не паниковать антиугоном.
- Сглаживание: медианный фильтр + отбрасывание точек с `hdop > 2` и скачков скорости.

## Антиугон (фаза 3)

Логика на ESP32 (работает даже без сервера, локально):
1. **Гео-забор:** задаём центр участка (точка станции) и радиус (напр. 50 м).
   Выход за радиус при `fix=1` И не в режиме «кошу/еду домой» → тревога.
2. **Lift + движение без команды:** подняли и понесли → тревога.
3. **Реакция:** ESP32 шлёт `state=error` + спец-топик `mower/mi302/alarm/theft`,
   E-stop ножа, громкий пьезо-сигнал, шлёт последние GPS-координаты пока есть питание.
4. **PIN:** снятие тревоги/старт после lift — только по коду из приложения (через сервер/BT).
5. Сервер: push-уведомление (HA companion app / Telegram) + лог координат.

```
mower/mi302/alarm/theft  →  { "ts":..., "lat":..., "lon":..., "reason":"geofence|lift" }
```

## Карта покрытия (фаза 3–4)

Источник — GPS-трек + одометрия колёс (Холлы), слитые простым фильтром:
- GPS даёт абсолютную позицию (±2–5 м), одометрия — гладкость между фиксациями.
- Сервер пишет трек в TimescaleDB/InfluxDB; «покошено» = буфер ширины 16 см вдоль трека.
- Визуализация: Leaflet/тепловая карта; «давно не кошено» → подсветка зон.
- Зоны/нет-зоны: полигоны на карте; пока без RTK это «мягкие» зоны (для аналитики/
  напоминаний), жёсткое удержание — по-прежнему провод.

## Путь к «кошению без провода» (RTK, фаза 5+, опционально)

Если захочется убрать провод и косить строго по карте:
- Нужен **RTK** (u-blox ZED-F9P ровер + база или NTRIP-коррекции) → точность ~1–3 см.
- Плюс контроль курса: магнитометр/IMU (BNO085) для удержания направления между фиксами.
- Это уже близко к **replace**-архитектуре (ESP32 сам рулит моторами по плану),
  поэтому решение принимаем после фазы 0 (см. [04-roadmap.md](04-roadmap.md), фаза 5).
- Риски RTK: деревья/стены → потеря фикса; нужен fallback (стоп/возврат к проводу).

## Почему не сразу RTK

- Дорого и сложно (база, антенны, открытое небо).
- Провод уже решает задачу удержания надёжно и бесплатно.
- «Максимально умно» на 80% достигается дешёвым GPS (трек+антиугон+карта),
  а оставшиеся 20% (кошение по карте) — отдельный большой апгрейд.
