# Архитектура «мозгов» и серверной части

Принцип: **пересадка мозга + слоистая переносимая связь**. ESP32-S3 сам читает
провод (matched filter) и сам рулит моторами; тело (силовые драйверы, сенсоры,
питание) от штатной платы переиспользуем. Со внешним миром ESP32 говорит на MQTT:
сегодня брокер слушает Home Assistant, завтра — собственный бэкенд, прошивку не меняем.
Алгоритмы движения — порт Ardumower Azurit (см. [06-open-source-landscape.md](06-open-source-landscape.md)).


## Почему MQTT в центре

- HA умеет **MQTT Discovery** — устройство само регистрирует свои сенсоры/кнопки.
- Свой бэкенд позже просто подписывается на те же топики — **миграция без переделки ESP32**.
- Брокер развязывает «робот» и «потребителей»: их может быть несколько (HA + бэкенд + Node-RED).

## Контракт MQTT (топики)

Базовый префикс: `mower/mi302/`

### Телеметрия (ESP32 → сервер), retained где уместно
| Топик | Payload | Частота |
|-------|---------|---------|
| `mower/mi302/state` | `idle\|mowing\|charging\|returning\|error\|lifted` | по событию |
| `mower/mi302/battery/voltage` | float, В | 10 с |
| `mower/mi302/battery/percent` | int, % | 10 с |
| `mower/mi302/battery/current` | float, А | 1 с при работе |
| `mower/mi302/motors/blade_current` | float, А | 1 с |
| `mower/mi302/sensors/rain` | `0\|1` | по событию |
| `mower/mi302/sensors/lift` | `0\|1` | по событию |
| `mower/mi302/sensors/tilt` | `0\|1` | по событию |
| `mower/mi302/sensors/bump` | `0\|1` | по событию |
| `mower/mi302/perimeter/inside` | `0\|1` (знак matched filter) | 1 с |
| `mower/mi302/perimeter/mag` | float (дистанция до провода, отн.) | 1 с |
| `mower/mi302/heading` | float, ° (IMU/одометрия) | 1 с |
| `mower/mi302/odom` | JSON `{dist_m, speed_l, speed_r}` | 1 с |
| `mower/mi302/gps` | JSON `{lat,lon,fix,sats,hdop,speed}` | 1–5 с |
| `mower/mi302/wifi/rssi` | int, dBm | 30 с |
| `mower/mi302/availability` | `online\|offline` (LWT) | — |

### Команды (сервер → ESP32)
| Топик | Payload | Действие |
|-------|---------|----------|
| `mower/mi302/cmd/start` | `PRESS` | начать кошение |
| `mower/mi302/cmd/stop`  | `PRESS` | стоп (мягкий) |
| `mower/mi302/cmd/home`  | `PRESS` | вернуться на станцию |
| `mower/mi302/cmd/estop` | `PRESS` | аварийная отсечка ножа+приводов |
| `mower/mi302/cmd/zone`  | `<zone_id>` | (фаза 3+) косить зону |

> LWT (Last Will & Testament): ESP32 публикует `availability=offline` при обрыве —
> сервер сразу видит «робот пропал».

## Пересадка мозга: ESP32 берёт управление телом

ESP32 не «нажимает кнопки» — он напрямую управляет **входами силовых драйверов**
моторов (PWM/DIR), которые мы определили в фазе 0. Штатный MCU из контура управления
движением **исключается** (его выходы на драйверы отключаются/перехватываются).
Логику движения (приём провода, PID, поведение, docking) ведёт сам ESP32 — это и есть
условие фикса «кручения» и «промаха» (см. [05-control-and-navigation.md](05-control-and-navigation.md)).

Что переиспользуем без изменений: силовой каскад драйверов, катушку провода
(берём её аналоговый сигнал), штатные датчики (rain/lift/tilt/bump/Холлы), питание/АКБ.

E-stop ножа — fail-safe: low-side MOSFET/реле в питании драйвера ножа, по умолчанию
выключено; ESP32 разрешает нож только когда `safe_ok`. Подробнее — [02-hardware-bom.md](02-hardware-bom.md).

## Локальная автономность (offline-first)

ESP32 не должен зависеть от Wi-Fi для безопасности:
- E-stop по lift/tilt дублируется **локально на ESP32** (а не только на сервере).
- Расписание кэшируется на ESP32; при потере связи косит по последнему плану.
- Сервер — это «ум» (погода, карты, аналитика), но не критичный путь безопасности.

## Слой данных для будущего бэкенда

- Временные ряды (АКБ, токи, GPS-трек) → **TimescaleDB / InfluxDB**.
- Карта покрытия — пост-обработка GPS-трека (см. [03-navigation-gps.md](03-navigation-gps.md)).
- API: REST для конфигурации/истории + WebSocket для live-телеметрии (re-publish из MQTT).
- Фронт: PWA (карта Leaflet + графики), позже — нативное приложение.
