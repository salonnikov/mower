# Реверс прошивки мейнборда косилки — GD32F305

Полный разбор оригинальной прошивки мейнборда (дамп `dist/gd32-mainboard-dump-v1.bin`, md5 `316927a42857b6f28bd7a0ad2d070de5`, снят по SWD). Анализ — radare2 (проект `tools/r2out/gd32.r2proj`). Все адреса — flash/RAM virtual; `fcn.080XXXXX` — авто-имена radare2.

> В прошивке сохранены `__func__`/путь-строки, поэтому восстановлены исходные имена модулей C (`process_docking_smooth.c`, `service_movement.c`, `service_config.c`, …). Реверс точный, не «вслепую».

---

## 0. Сводка для нетерпеливых

- **Чип:** GigaDevice GD32F305 (ARM Cortex-M4F, 120 МГц), 1 МБ внутренней флеши. **ОС: FreeRTOS** + объектный C с vtable.
- **Образ = загрузчик (96 КБ @`0x08000000`) + приложение (@`0x08018000`, ≤928 КБ).** Перед запуском приложение проверяется по CRC; не сходится → ребут в загрузчик.
- **Почему мажет мимо базы (2–5 см):** финальный заезд **разомкнутый**, без точного датчика. Близость к станции — по амплитуде сигнала провода (`< 30999`), решающий поворот на контакты — **со случайным углом 45–125°** (`rand`), успех судится **только по току зарядки**. Подрулить вбок нечем. → структурно неизбежный промах.
- **Почему «крутится на месте»:** провод = **булев флаг на катушку** (0/1/2), развороты — **рандомные углы**, исполняются **разомкнуто в тиках**, IMU-курс в обратной связи не участвует. На граничном сигнале классификатор мерцает → `cycle left ↔ right` каждый тик → осцилляция.
- **Патч реален:** правишь байты приложения `0x08018000+`, пересчитываешь GD32 CRC-32, пишешь в env-ключ `MB_CRC` (EasyFlash) — образ принимается. CRC-гейт + staging = не кирпичит.
- **MQTT в мейнборде НЕТ** — это работа ESP дисплея. Мейнборд говорит только `&{…}<csum>#` по UART.

## 1. Архитектура

### ОС и структура
**FreeRTOS** (heap_4, порт ARM-CM4F) — подтверждено `__FILE__`-строками `..\src\platform\freertos\{tasks,queue,port,heap_4}.c`. Слои:
- `src/platform/freertos/*` — RTOS
- `src/platform/driver/driver_*` — драйверы (`driver_battery_snk_v1/_v2.c`, `driver_dpport/bdport/ledport.c`, `driver_mems_snk_v13.c`)
- `src/framework/service_*` — сервис-объекты с vtable (`service_bms`, `service_config`, `service_display`, `com_easyflash`)
- `src/app/process/process_*` — обработчики верхних состояний (по .c на состояние)

Объектный C: глобальные структуры с указателями на функции, вызовы через `blx [rN+off]` (process-manager @`0x200000a8`, BMS @`0x20000350`).

### Boot (reset `0x08011a3c`)
SP=`0x20016ed8`. Init (`fcn.08011a3c`): тактирование/GPIO (`fcn.08010cd0`) → проверка валидности/релокейт векторов → канарейки стека FreeRTOS → периферия+**watchdog** (`fcn.08011dfc`) → `fcn.080003bc`: включает **FPU** (`CPACR|=0xF00000`) и уходит в `xPortStartScheduler` (планировщик забирает управление).
> Примечание: reset `0x08011a3c` лежит в области **загрузчика** (<`0x18000`). У приложения свой вектор по `0x08018000`. Код приложения (process_*, докинг, навигация) — в `0x0802xxxx–0x0808xxxx`.

### Задачи FreeRTOS
process-manager (мастер-машина), bms, map, highspeed-loop, time, fuse, mems, и три порта: **dpport/bdport/ledport** (init/receive/send-задачи). Очереди/мьютексы: «left/right wheel speed queue», «speed data queue», `motor_spi_mutex`, `control mutex`, `dpport send lock mutex`, `pnt_queue` (картирование).

## 2. Машина состояний (мастер-контроллер)

**`set_main_state(state)` = `fcn.0807976c`** — единственная точка записи (76 вызовов), клампит `0..10`, пишет в **RAM `0x200000bc`** (=`0x200000ac+0x10`). Это ровно то, что мы снифали по UART.

| Состояние | Имя | Доказательство |
|---:|---|---|
| 0,1,2 | boot/init | `fcn.08054a16: movs r0,2` |
| 4 | wait (подсостояние) | `process_wait.c fcn.080270a8` |
| 5 | transitional/leave | departure `@0x803a40e` |
| 6 | **idle/готова** | `@0x8039280/0x803a340 movs r0,6` |
| 7 | **error** (+код ошибки) | `@0x803a3a6 movs r0,7` |
| 8 | departure→**кошение** | `@0x80273e8/0x803a586 movs r0,8` |
| 9 | возврат/докинг | `process_docking_smooth.c @0x803cb6c movs r0,9` |
| 10 (0xA) | в доке/**зарядка** | `process_charging.c fcn.08025c2e: movs r0,0xa` |
| 11 | стоп/пауза | UART-уровень (кламп отвергает 11 как main_state) |

Переходы логируются через `fcn.08048adc(level,module,file,line,fmt,…)`. Примеры: idle→cut `«idle manual start command…change to cut»`; idle→dock `«dock command and on border, change to dock»`; idle→error `«start command, but no border, change to error»`; cutting→error `«change to error slope over»`.

## 3. Навигация / следование по проводу / развороты

Модули: `service_border.c` (сенсинг), `service_movement.c` (углы→тики), `process_cutting.c`, `process_departure_smooth.c`, `process_wait.c`.

### Как чувствует провод
Сигнал зарытого провода обрабатывается матч-фильтром в `service_border.c` и наружу отдаётся как **дискретный флаг на катушку**: `0=нет сигнала, 1=внутри, 2=снаружи`. **Две катушки** (лево `[ctx+0x24]`, право `[ctx+0x2c]`). Вся навигация построена на 4 комбинациях (обе-внутри / лево-вне / право-вне / обе-вне / потеря). Плюс величина силы (`strength l=%d,r=%d too low`).
- Резолвер направления: **`fcn.0802c900`** + `fcn.0802c424`.
- Гейт безопасности `fcn.080270a8` (`autowork check safety`) не пускает в кошение при невалидном сигнале (`border sinal not correct`).

### Направление и угол разворота
Направление — по тому, какая катушка вышла за провод (`right out→cycle left`, `left out→cycle right`, …). **Угол — РАНДОМНЫЙ**, из `rand_range` (`fcn.0804f2c8`, **79 вызовов**):
- `rand_range(120,150)°` `@0x0802cce2`
- `rand_range(90,135)°` `@0x0803024e`
- `rand_range(60,100)°` `@0x080302f2`
- фикс. recover-углы 45°/135° (`@0x0802cf18/ceaf`), квант `0x2d0=720`.

Поворот исполняется **разомкнуто, в тиках энкодера**: `angle_to_cycle_tick` (**`fcn.08020520`**) = `угол×697/K`, K — пер-модельная константа (поле `+0x32…+0x52`). **IMU-курс в обратной связи разворота НЕ участвует** (только уклон + карта).

### Recover-поведения (`fcn.080321d4`, 11.5 КБ)
| Триггер | Детект | Манёвр |
|---|---|---|
| HIT | контактный | назад + рандом-угол `cycle` |
| LIFT | концевик подъёма | назад+поворот; дебаунс `serial/lift count` |
| SLOPE | IMU pitch/roll | реверс-цикл, смена на slope-speed |
| STALL | ток/энкодер | назад+поворот, тормоз ножа |
| ULTRASONIC | УЗ-дальномер | назад+поворот (без рулёжки по дистанции) |
| SLIP/overtime | проскальзывание тиков | `back slip overtime, cycle dir` |

Эскалация: «finish but still hit/ultrasonic → reverse cycle to find»; не нашёл провод → **`random move`** (`fcn.08077e10`).

### Почему «крутится на месте»
1. Все углы рандомны, цели по курсу нет → неэффективный отскок (by design).
2. На грани сигнала классификатор катушек мерцает in/out → `fcn.0802c900` чередует left/right короткими квантами → платформа осциллирует ≈ на месте.
3. **Необработанные комбинации** (`border left no signal right in, undealed`) — мёртвые ветки, команда не выдаётся, тот же ввод → петля.
4. Recover→re-find реверсит сторону каждый раз, пока сигнал не прочитается чисто → топчется.

Корень: **разомкнутые тиковые развороты + рандом + булев сигнал, без PID по курсу.**

## 4. 🎯 Докинг — почему ВСЕГДА мажет мимо базы

Модуль `process_docking_smooth.c`, мастер `fcn.08044188` (15.6 КБ). Структура докинга @`0x20000098`.

### Последовательность
- **A. Решение идти на базу** (диспетчер `aav.0x08065a08`): `«bat low=%dmV and on border, change to dock»`; также по расписанию/дождю/износу. Сначала надо **поймать провод** (докинг — относительно провода, не станции).
- **B. Едет по проводу к станции** (`docking_step_follow_boundary`): **bang-bang** «оседлай провод» — `in border→right cycle`, `out border→left cycle`. Скорость через `dock_is_near_station`/`fd_is_near_station`: near→slow, else→follow. Контура рулёжки нет, только пресеты скорости 0–4.
- **C. «Близко к станции»** (`fd_is_near_station` `fcn.0804b0b8`): **обе магнитуды сигнала провода `< 30999 (0x7918)`**. Разрешение — десятки см. **Нет маяка, ИК, датчика касания, дифф-наведения.**
- **D. Съезд с провода на контакты** (`docking_step_leave_border`): `rand_range(45,125)` → **поворот под СЛУЧАЙНЫМ углом 45–125°** и вперёд «примерно туда».
- **E. Прибытие/таймаут**: `is_reach_charger_station` (`fcn.0805e194`) = **один булев** флаг. `is_on_station_overtime` = счётчик vs `0x140 (320)`/`0x1E0 (480)`.
- **F. Проверка подключения** (`docking_step_check_station` `fcn.08043bfc`) — **ТОЛЬКО по току зарядки**: `current normal→start charging`; `too small→back to reconnect`; `too high→error 0x2000`; `over times→error`. Ретраи: `[+0x60]==5`.
- **G. Сел→зарядка** (`process_charging.c`).

### Корень промаха
**Финальное позиционирование РАЗОМКНУТО, обратной связи нет:**
1. **Датчика докинга в прошивке нет** — близость только по амплитуде провода (порог `30999`), грубо.
2. **Решающий поворот — рандомный** (`fcn.0804f2c8(0x2d,0x7d)` в `fcn.08044188`) → точка приземления недетерминирована.
3. **«Доехал/подключился» — булево по току**; промахнулся → не знает, в какую сторону → **не подруливает**, просто `reconnect` тем же рандомным манёвром (до 5 раз → `error`).

> ⚠️ Те «два датчика у порта дока» прошивка **для наведения не использует** — успех только по току. Похоже, это зарядные контакты, не хоминг-сенсоры.

### Рычаги починки
| Параметр | Значение | Где |
|---|---|---|
| **Угол съезда (рандом)** | **45–125°** `rand_range(0x2d,0x7d)` | `fcn.08044188 ~0x803fd96` |
| Порог «near station» | **30999 (`0x7918`)** | `fcn.0804b0b8 @0x804b11a` |
| Дебаунс near | 10→30 `[+0x7d]` | `fcn.08032e20` |
| On-station overtime | 320/480 | `fcn.0805e02c` |
| Connect timeout | 400 `[+0x46]` | `fcn.08043bfc` |
| Ретраи reconnect | 5 `[+0x60]` | `fcn.08043bfc` |

**Главный рычаг:** заменить рандом-угол на **фиксированный калиброванный** (+ поправка на замеренный сдвиг) → убирает разброс. Дополнительно — порог `0x7918` (заходить прямее/медленнее) и дольше «дожимать». По-настоящему точно — только добавив хоминг-сенсор + контур в `docking_step_follow_boundary`/`_leave_border`.

## 5. Батарея / BMS

Модули `process_charging.c`, `service_bms.c`, `driver_battery_snk_v1/_v2.c`. Ctx зарядки @`0x200000a0`, данные BMS @`0x20000350`.

- **Защита ячеек** (`fcn.080279ac`, дебаунс 10): over-discharge `min<[+0xc4]`→`[+0x14]=3` (гистерезис +150 мВ); over-charge `max>[+0xc6]`→`[+0x14]=2` (−100 мВ). Поле `[+0x14]`: 1=норма,2=перезаряд,3=переразряд.
- **Темп-компенсация потолка заряда** (`fcn.08038c48`): ≥45°C→4100 мВ; 10–45°C→4200 мВ; ≤10°C→4120 мВ.
- **Подсостояния зарядки** (`aav.0x08065a08` по `[+0x18]`): 1/2 трикл/CV (`fcn.0807ff04`), 3 CC (`fcn.08025fd4`), 4 temp-protect (`fcn.080265dc`). Сеттер шага `fcn.080267f8`. Полный заряд `fcn.08025c90` (флаг держится >80 циклов).
- CC-переходы: ток мал ≥80 циклов → idle(6) если %>6 иначе error(7); ток высок >40 → error `0x2000`. «charging finish by schedule→idle».
- Низкий заряд: флаг @`0x200000ae`; `process_wait.c` → `«battery voltage low, goto charging»` → `set_main_state(8)` (уехать к доку).

## 6. Расписание / авто-кошение

Логика в `process_wait.c` (`fcn.080270a8`, `check_auto_work`), недельная таблица (Start/End/Duration/Border-Cut, заголовок лог-экспорта `@0x080165a8`). Поддержка дней типа **trim** (кромка) и **auto**. Пустое расписание → ручной режим. Триггер: время в окне → `«Robot on schedule, start work»` → гейты (дождь/безопасность/батарея) → `set_main_state(8)` → departure → кошение. Дождь: отдельный `rain_delay`, прерывает выезд (`«departure rain…dock»`).

## 7. Протокол UART (три JSON-порта)

Мейнборд держит **три структурно одинаковых JSON-линка**, общая библиотека кадрирования `deal_message.c` + cJSON:

| Порт | Файл | С кем | Диспетч |
|---|---|---|---|
| **дисплей (ESP)** | `driver_dpport.c` | плата дисплея — **наш снифнутый JSON** | `fcn.080470f0` |
| **граница/навигация** | `driver_bdport.c` | плата сигнала провода | `fcn.08063880` |
| **LED** | `driver_ledport.c` | плата индикации | `fcn.0806396c` |

> Имена `bdport`/`dpport` в коде vs «какая плата» — есть неоднозначность (встречается и `dpport_border_offset`). Достоверно: снифнутый дисплейный линк диспетчеризуется `fcn.080470f0`, статус строит `fcn.08081300`.

### Кадр и CRC
`&` (0x26) + JSON + `<csum:1B>` + `#` (0x23). **CRC-8 Dallas/Maxim** (poly `0x8C`, init 0), `fcn.080241d0`. TX-фреймер `add_crc_and_packed` `fcn.0801fb80`; RX-дефреймер+проверка `deal_receive_message` `fcn.0806e938`.

### Диспетчеризация RX (`fcn.080470f0`)
`cJSON_Parse` → `GetObjectItem("cmd")` → `category = cmd & 0xFF000000` → линейный поиск в 16-записной таблице (`[+0x60]=category,[+0x64]=handler`, регистрация `fcn.08020036`) → `blx handler(obj)`. Конфиг-кадры обрабатывает `service_config.c` `fcn.08076b2c` (~120 ключей: `robot_name,sn,sch_en,rain_en,zone_en,region,pwd_en,pwd_rst,run_param`…).

### Таблица команд (TX = мейнборд→дисплей)
| cmd | Имя | Строитель | Эффект |
|---|---|---|---|
| `0x33000020` | STATUS | `fcn.08081300` | полный снимок состояния |
| `0x33000021/22/26/27/28/30` | info/voltages/schedule/rain/zones/map | пул `0x8081xxx`/`fcn.080810e0` | телеметрия/настройки |
| `0x40000011` | heartbeat+rtc | `fcn.0802244c` | пульс+время |
| `0x41000002/03` | lock/unlock | `fcn.0803c4e8` | экран замка |
| `0x41000004` | err:N | `fcn.0803baa0` | код ошибки |
| `0x41000006` | **HOME** | `fcn.0803ba68` | запрос «на базу» |
| `0x41000007` | docking | `fcn.0803ba30` | докинг |
| `0x41000020` | PIN result | `fcn.08072400`(=1)/`fcn.08072304`(=0)/`fcn.08072134`(lockout) | вердикт PIN |
| `0x50000021` | battery | `fcn.0803b85c` | `bat:N` |

TX-строители вызываются не входящей командой, а экранной машиной `fcn.0806db88` (читает screen-event из `[0x200003b0+0xc]`, `tbb`-диспетч 0..0x11).

### STATUS-поля (`fcn.08081300`, cmd 0x33000020)
`state←[r4+0x24]`, `station(bool)←[+0x2c]`, `border_state←[+0x1a]`, `error←[+0x19]`, `on_minutes←[+0x1b]`, `total_minutes←[+0x3c]`, `cur_minutes←[+0x1c]`, `bat_max_temp←[+0x1e]`, `work_area←[+0x38]`, `current_area←[+0x44]`, `bat_health←[+0x48]`, `rain_delay←[+4]`, `stop_state←[+0x14]`.

### MQTT
**В прошивке мейнборда MQTT/broker/topic НЕТ** (единственное «mqtt» — в HTML лог-странице). Мейнборд говорит только этот JSON; MQTT-мост — целиком на ESP дисплея.

## 8. PIN / безопасность

PIN **не зашит** — хранится в NVM-ключе **`pwd`** (4 байта), флаг включения `usr_pwd_en`. Загрузка: `load_user_password` `fcn.08060d00` → RAM `0x2000027c`. «0000» = заводское значение поля `pwd`, меняется через `service_config.c` (`fcn.08076b2c`, ключи `pwd_en/pwd_rst`).

Поток: мейнборд шлёт `0x41000002` lock → дисплей собирает цифры, шлёт на мейнборд → машина безопасности `fcn.08054fe4` сравнивает с `pwd`, считает ошибки с **локаутом** (`«input error password count…over times, lock and wait»`) → вердикт `0x41000020` (`result:1/0`/`left_time`). Верно → `0x41000003` unlock.

> Модель доверия: PIN ходит по UART **открытым текстом** (только CRC-8 для целостности, не аутентификация) — любой на шине видит цифры и вердикт.

## 9. Загрузчик / обновление / CRC / watchdog

### Карта внутренней флеши (1 МБ)
- `0x08000000–0x08018000` — **Загрузчик (BTL)**, 96 КБ.
- `0x08018000–0x08100000` — **Приложение (MB)**, макс **`0xE8000` (928 КБ)**.
- **HW CRC-32** @`0x40023000` (poly `0x04C11DB7`, init `0xFFFFFFFF`).

### Типы образов (`SetFirmInfo` `fcn.0800214c`)
MB(1)/BB(2)/DB(3)/BTL(4)/LB(5)/DB-L(6) — каждый со staging-областью в **внешней SPI-флеши** и env-ключами `*_SIZE/*_CRC/*_BVER/*_BRF`. Под-платы (BB/LB/DB) — **внешние МК, шьются по serial** (`programBB` `fcn.08006658`).

### Самопроверка MB по CRC (гейт для патча) — `fcn.08006fe8`
Читает env `MB_SIZE/MB_CRC/mb_sv/MB_BVER` → `crc_feed_words(0x08018000, MB_SIZE>>2)` → сравнивает с `MB_CRC` (3 ретрая). Не сошлось → лог `«CRC check error»` → 5 с → reset в загрузчик.

### 🔧 Как залить патч (не кирпичит)
1. Изменить байты в `0x08018000 .. +MB_SIZE`.
2. Пересчитать GD32 HW CRC-32 (poly `0x04C11DB7`, init `0xFFFFFFFF`) над `MB_SIZE & ~3`.
3. Записать новое значение в **env-ключ `MB_CRC`** (EasyFlash; при изменении размера — и `MB_SIZE/MB_BVER`).
Иначе verify падает и юнит ребутится в загрузчик. Есть OTA-канал (`fcn.08007e30`) и заводской откат (`cfg_rst==0xA5A5`, `BACK_TO_FACTORY==0xA5`).

### Env = EasyFlash (armink) на внешней SPI-флеши
`«EasyFlash V%s … success»`, `ef_port.c`, backup-env переживает плохие записи. Конфиг-версия: `«config version=%d<20302, erase exflash»`.

### Watchdog
**IWDG (FWDGT) @`0x40003000` активен.** Конфиг `fcn.0800c35c`, enable `fcn.0800c3bc` (`0xCCCC`), feed `0xAAAA` (`fcn.0804bca0`/`fcn.0800c3ac`). Кормят **разные задачи** (`0x08049406`, `0x0805e352`, `0x08064656`, `0x0807ef80`, зарядка…). Поэтому **halt по SWD остановил всех кормильцев → IWDG сбросил чип** (наш ребут). WWDG не используется. Причина сброса логируется (`«watchdog Triggered»`).

## 11. Что с этим делать (варианты)

1. **Патч докинга в прошивке** (самый прямой к твоей боли): в `fcn.08044188` заменить `rand_range(45,125)` на фиксированный калиброванный угол → пересчитать `MB_CRC` → залить по SWD. Убирает разброс «мимо базы».
2. **Тюнинг навигации**: сузить рандом-диапазоны углов, поднять порог силы сигнала, поправить `K` (угол→тики) под шасси — меньше «кручения».
3. **Включить `quick_dock`/проверить опции** через env (`set_product_enable_*`).
4. **Внешний контур (наш ESP)**: добавить хоминг-сенсор и доводку на финале докинга — единственный путь к суб-см точности (прошивка этого не умеет в принципе).

## Приложение — ключевые адреса
- Машина состояний: `set_main_state` `fcn.0807976c`, поле `0x200000bc`, low-batt `0x200000ae`
- Boot: reset `fcn.08011a3c`, scheduler `fcn.080003bc`, периферия/WDT `fcn.08011dfc`
- Навигация: резолвер `fcn.0802c900`, recover `fcn.080321d4`, угол→тик `fcn.08020520`, рандом `fcn.0804f2c8`, follow `fcn.0803a7a0`, random-move `fcn.08077e10`
- Докинг: мастер `fcn.08044188`, check-station `fcn.08043bfc`, near `fcn.0804b0b8`, reach `fcn.0805e194`, overtime `fcn.0805e02c`
- Батарея: dispatch `aav.0x08065a08`, CC `fcn.08025fd4`, защита `fcn.080279ac`, BMS-task `fcn.08038c48`; ctx `0x200000a0`, данные `0x20000350`
- Протокол: RX-диспетч `fcn.080470f0`, статус `fcn.08081300`, CRC-8 `fcn.080241d0`, фреймер `fcn.0801fb80`, дефреймер `fcn.0806e938`, конфиг `fcn.08076b2c`
- PIN: `load_user_password` `fcn.08060d00` (NVM `pwd`→`0x2000027c`), машина `fcn.08054fe4`
- Загрузчик: `SetFirmInfo` `fcn.0800214c`, MB-CRC `fcn.08006fe8`, HW-CRC `0x40023000`, OTA `fcn.08007e30`; MB@`0x08018000` (≤`0xE8000`), BTL@`0x08000000`
- Watchdog: IWDG `0x40003000`, feed `fcn.0804bca0`
