# Реверс второго чипа (chip2) + вердикт по multitrack

Дамп `dist/gd32-mainboard-chip2-dump-v1.bin` (md5 `ddbc6b5aea44252eef3c6a63c39d2010`, 256 КБ, сверен двумя чтениями). Проект radare2 `tools/r2out/chip2.r2proj`. Reset `0x080001b4`. Версии: HW 230500, SW 50003.

## 0. Что такое chip2
**Второй GD32F30x (Cortex-M4, FreeRTOS) — плата обработки сигнала провода периметра.** Чистый датчик, **моторами не рулит** (моторы на chip1). Семплит две носовые катушки, считает силу сигнала, классифицирует in/out и шлёт «border-сообщения» на chip1 по USART2. Фолтит на SWD-halt (SysTick-busy-wait + единственный USART2-DMA) → читать только без halt.

## 1. Носовые катушки = входы chip2
Два индуктивных коила (KCD KD-002, `LEFT`/`RIGHT`, в носу у порта зарядки, разъём **J9**) → синхронный двойной АЦП:
- **ADC0 канал 5 = LEFT**, **ADC1 канал 9 = RIGHT** (`fcn.08019bf4`), sample time 7 (макс SNR), синхронный parallel-режим.
- Запуск от **TIMER1 TRGO** (база `0x40000000`, PSC=0, ARR=0xC7), **DMA0** льёт `ADC0_RDATA` в буфер **800 отсчётов** (`[obj+0x92]=0x320`).
- Базовая линия (DC-смещение) калибруется (`fcn.0801a1f8`, среднее 8, хранится `0x2000004c`, лог `base voltage=%d`).

## 2. Обработка волны → непрерывная магнитуда
Детект **временной**, не FFT/тон (строка `Frequency Test` — это тест ТАКТОВОЙ частоты, не сигнала):
- **Сегментация по нулю** (`fcn.08015dcc`): режет буфер по сменам знака, допуск зазора 50.
- **Фичи сегмента** (`fcn.08015c68`): ширина `(end-start+1)` (band 4..15), и **пиковая амплитуда** (min/max int16) = **`str`** — знаковая непрерывная магнитуда (монотонна по расстоянию до провода; **знак = сторона провода**).
- **Классификатор** (`fcn.080159a0`) сверяет последовательность сегментов с шаблоном кодированного пакета, возвращает 0..4 (допуски 10/50/150). То есть сигнал провода — **кодированный импульсный пакет** (ref-период `0x49b4`).
- Коды состояния катушки: **0=нет сигнала, 1=внутри, 2=снаружи** (+промежуточные 3/4/6/7/8). Хранятся в под-структурах `0x20000078→[+4]+0x38…`: `[+1]`=состояние, `[+4]`=знаковая магнитуда.

**🎯 Главное:** магнитуда **непрерывная** и **дифференциальная** (лево/право), и она **уже передаётся** на chip1 (см. ниже) — не только бит in/out.

## 3. Протокол chip2 → chip1 (border-сообщения)
**Голый cJSON по USART2, БЕЗ рамки и БЕЗ CRC** (в отличие от дисплейного `&{…}<crc>#`). Граница больше сообщения = длина DMA. Строитель `send_border_message` (`0x080195c4`):
```
{"cmd":16,"ls":<num>,"rs":<num>,"lf":<num>,"sls":<num>,"srs":<num>}
```
- `ls`/`rs` = **левая/правая непрерывная знаковая магнитуда** (основные коилы).
- `sls`/`srs` = вторичные/«search» магнитуды.
- `lf` = флаг/уровень (lift). Доказано: значения идут через `cJSON_AddNumberToObject` как числа (int→double), знак = in/out.
- `{"cmd":26}` = follow/search-триггеры (без данных). Плюс version-кадр (`hv`,`sv`).

**chip2 НИЧЕГО не парсит от chip1** (нет `cJSON_Parse`/`GetObjectItem`; RX-IRQ USART2 — только транспорт, callback не подключён). **Линк односторонний: датчик→мозг.**

## 4. Импичмент/инъекция
Раз кадры — plain JSON без CRC/auth/последовательности, **малина может притвориться платой-датчиком** и слать chip1 поддельные `ls/rs/lf` → его навигация среагирует. Но это **манипуляция его же (рандомной) логикой**, не прямые команды движения. Также можно просто **слушать** этот UART → получать непрерывную боковую позицию даром.

## 5. Вердикт по `multitrack`/`smooth`/`multizone` (chip1)
**Систематического кошения полосами/спиралью в прошивке НЕТ, и флагом не включается.** Внутренность газона **всегда** рандом (целочисленная машина `fcn.080321d4`, **ноль тригонометрии** во всей cutting-зоне `0x0802bc00–0x08033000`, нет heading-hold, нет слов `lane/serpentine/sweep/row/spiral`).
- **`multitrack`** (`mtrack_en`, config `0x20000209`) — **инертный stub**: байт включения + счётчик, которые **никто из логики движения не читает** (только лог + геттер). Углы поворота — фиксированные дискретные (30/60/100/135° из `cut_get_suggest_angle` `fcn.0802c2cc`).
- **`smooth`** (`smooth_en`, `0x20000213`) — меняет **только манёвр выезда с базы и заезда на док** на следование по проводу с настраиваемым **`border_offset`** (через co-processor «dpboard»/dpport). Внутренность газона — по-прежнему рандом. **Реальный, но узкий «бонус аккуратности» — включается `set_product_enable_smooth 1`, переживает ребут, низкий риск.**
- **`multizone`** — делит сад на **до 4 пропорциональных подзон** (4 точки + проценты + pass/ex_pass-расписание `set_autozone_point` `fcn.0807631c`); внутри каждой — рандом. `service_workmap.c` строит карту периметра (углы, самопересечения, фиттинг линий) — для докинга/зон, не для рядов.

Включение флагов = запись в NVM (ключ `rb_en_mag`), обратимо; делать по одному с сохранением исходных значений.

## 6. Вывод для навигации
**Железо уже правильное:** 2 дифференциальных носовых коила → непрерывный L/R-сигнал провода (`ls/rs`), одометрия (холлы), IMU (курс), моторы на chip1. **Прошивка это тратит впустую** (рандом + порог + случайный угол на докинге). Ни один флаг не даёт ровных полос. Значит нормальная навигация = **новая логика**, и её естественный дом — **патч chip1** (там уже есть все нужные данные: `ls/rs`, одометрия, IMU, управление моторами). RPi возможна как планировщик, но всё равно потребует, чтобы chip1 принимал команды (патч). Подробности стратегии — обсуждаются.
